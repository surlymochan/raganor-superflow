# Superflow 阶段 1：Clarify（澄清）

**命令**: `/superflow:clarify <一句话需求>`

**解决的问题**: 需求模糊、不完整

## 流程

### 1. 并行调研（Background）

启动两个并行调研任务：

**explore 调研**:
```
task(
  subagent_type="explore",
  run_in_background=true,
  prompt="探索当前项目结构，了解：
  1. 项目技术栈（package.json / requirements.txt 等）
  2. 现有代码组织方式
  3. 相关的现有模块/功能
  4. 编码规范和模式
  
  目的是理解需求落地的上下文。"
)
```

**librarian 调研**（如涉及外部技术）:
```
task(
  subagent_type="librarian",
  run_in_background=true,
  prompt="查找关于 [具体技术/框架] 的最佳实践：
  1. 官方文档推荐的使用方式
  2. 常见错误和陷阱
  3. 测试策略
  4. 性能考虑
  
  目的是给出可靠的技术方案建议。"
)
```

### 2. 渐进式澄清

收集调研结果后，一次问一个问题：

**问题模板**:
1. "这个功能的核心用户是谁？"
2. "解决什么具体问题？"
3. "成功的标准是什么？"
4. "有什么技术约束？"
5. "优先级是什么？（功能完整性 vs 速度）"

### 3. 方案对比

提出 2-3 个方案，格式：

```markdown
## 方案对比：<需求>

### 方案 A：[名称]
**优点**: ...
**缺点**: ...
**估计工时**: ...

### 方案 B：[名称]
**优点**: ...
**缺点**: ...
**估计工时**: ...

### 推荐：方案 [X]
理由：...
```

### 4. 创建 Spec 草稿

```markdown
# Spec：<功能名称>

**版本**: 0.1 (草稿)
**日期**: YYYY-MM-DD

## 1. 目标
[一句话描述要解决的问题]

## 2. 范围
### 必须做
- [ ] ...
### 不做
- [ ] ...

## 3. 接口设计
[API/函数签名]

## 4. 数据模型
[数据结构]

## 5. 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

## 产出

```
openspec/changes/<name>/
├── spec.md       # Spec 草稿
├── context.md    # 调研结果
└── options.md    # 方案对比
```

## 门禁

**用户必须**：
1. 选择一个方案
2. 批准 Spec 草稿

## 下一步

批准后，执行：`/superflow:plan <name>`
