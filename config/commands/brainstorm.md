# Superflow 阶段 1：脑暴

**用法**: `/superflow:brainstorm <需求>`

## 流程

1. **澄清需求** - 一次一个问题，理解目的、约束、成功标准
2. **提出 2-3 个方案** - 对比优缺点，给出推荐
3. **呈现设计** - 架构、组件、数据流、错误处理，逐节确认
4. **创建 PRD 草稿** - `openspec/specs/domain/<domain>/prd-<name>.md`
5. **创建 Spec 草稿** - `openspec/changes/<name>/proposal.md`

## 产出

- `openspec/specs/domain/<domain>/prd-<name>.md` - **产品需求文档 (PRD)**
- `docs/plans/YYYY-MM-DD-<name>-design.md` - 设计文档
- `openspec/changes/<name>/proposal.md` - 变更提案

## PRD 模板

```markdown
# <名称> 产品需求文档

**版本**: 0.1 (草稿)
**日期**: YYYY-MM-DD
**状态**: 草稿/审查中/已批准

## 1. 产品概述

### 1.1 问题陈述

用户面临什么问题？为什么需要这个功能？

### 1.2 目标用户

谁是目标用户？他们的痛点是什么？

### 1.3 成功指标

如何衡量这个功能的成功？(KPI/OKR)

## 2. 需求范围

### 2.1 必须做 (Must Have)

- [ ] 核心功能 1
- [ ] 核心功能 2

### 2.2 应该做 (Should Have)

- [ ] 重要功能 1
- [ ] 重要功能 2

### 2.3 可以做 (Nice to Have)

- [ ] 可选功能 1

### 2.4 不做 (Won't Have)

- 明确排除的功能

## 3. 用户故事

| ID | 作为 | 我想要 | 以便 | 优先级 |
|----|------|--------|------|--------|
| US1 | 用户 | ... | ... | P0 |

## 4. 功能需求

### 4.1 功能 1

**描述**: ...

**验收标准**:
- [ ] 标准 1
- [ ] 标准 2

## 5. 非功能需求

- 性能要求
- 安全要求
- 可用性要求

## 6. 依赖与风险

| 类型 | 描述 | 缓解措施 |
|------|------|----------|
| 依赖 | ... | ... |
| 风险 | ... | ... |

## 7. 等待批准

阶段 1 完成，等待用户批准。
```

## 批准后

用户批准设计后，进入阶段 2：`/superflow:plan`
